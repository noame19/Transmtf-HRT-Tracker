# 用药提醒防错触交互设计

## 目标

把"该用药弹窗"、"过期补打卡弹窗"、"用药页 banner"三个地方的选项按钮从"点一下就生效"改成"点两下才生效"，避免误触带来的用药安全风险（错以为自己吃了实际没吃 / 错以为自己跳过了实际没跳）。

旧的二次确认走的是"弹窗的弹窗"（全屏独立 confirm 弹窗盖在提醒弹窗之上），体验差且会引发"画面闪一下"的视觉割裂。本设计用"按钮原地等待确认"模式替代，0 弹窗、0 状态切换。

## 适用范围

涉及三个地方：

| 叫法 | 组件 | 状态 | 按钮集 |
|---|---|---|---|
| 该用药弹窗 | `ReminderModal` | on_time | 已服用 / 推迟1天 / 推迟2天 + 右上角 X |
| 过期补打卡弹窗 | `ReminderModal` | late | 已服用 / **跳过本次** / 推迟1天 / 推迟2天 |
| 用药页 banner | `ReminderBanner` | 永远 | 已服用 / **跳过本次** / 推迟1天 / 推迟2天 |

**不在范围内**：

- `DoseFormModal`（补登用药表单弹窗）—— 它的"保存 / 删除 / X"是表单编辑场景，跟提醒的"防错触"不同类
- 任何通知权限 / 公告类弹窗

## 核心交互机制

### 视觉基线

| 状态 | 背景 | 文字 | 描边 |
|---|---|---|---|
| 默认（未点过 / 切换时上一选项） | 透明 | 原色（`var(--text-primary)`） | 1px 普通（`var(--border-primary)`） |
| 第一次点击后（等待确认） | 主题色（`useTheme().colors[500]`） | 原色（**不变**） | 无（去掉描边） |
| 第二次点击同一位置 | 闪一下勾动画 → 关闭 | — | — |

**"已服用"按钮降级**：当前是"白字 + 主题色填充"的主按钮样式（`glass-btn-primary`）。本次改造把它降级为"原色文字 + 透明背景 + 普通描边"，跟其他三个选项视觉完全平等。理由是"加二次点击确认后不再是点一下就生效"，不应再享主按钮的视觉权重。

### 状态机

每个弹窗 / banner 维护一个**当前在等待确认的按钮标记**（同一个弹窗内同一时刻只有一个按钮处于等待确认状态）：

- 用户点按钮 X：
  - 如果当前没有等待中的按钮 → 标记变成 X，按钮变主题色背景（进入等待确认）
  - 如果当前等待的是 X → 调用 X 的处理函数，标记清空
  - 如果当前等待的是 Y（不是 X）→ 标记从 Y 切到 X，Y 恢复默认样式，X 变主题色背景

**没有时间窗口**。等待确认状态一直保持，直到用户点同一按钮（触发）或点别的按钮（切换）。

### "跳过本次"特殊处理

只有过期补打卡弹窗的"跳过本次"按钮需要特殊处理（因为它是"偏离计划"的危险操作）：

- 第一次点 → 按钮变主题色背景 + **按钮下方展开一段警告纯文本**（没有按钮）
- 警告文本内容：沿用现有 `t('reminder.banner.skip_confirm.body')`，即"将跳过今日原有计划，原计划不会顺延。强烈影响身体激素状态，您确定吗？"
- 第二次点同一"跳过本次"按钮 → 视为确认 → 触发跳过
- 切换到其他按钮（点已服用 / 推迟1天 / 推迟2天）→ 警告文本自然淡出，旧按钮恢复默认，新按钮变主题色背景
- 警告文本视觉：极淡玫红底（`rgba(244, 63, 94, 0.06)`）+ 玫红字（`var(--text-soft-rose)`），与按钮等宽，无操作按钮
- 出现动画：淡入 + 自上滑下 6px（200ms 缓出）
- 消失动画：淡出（100ms）

用药页 banner 的"跳过本次"按钮**保留**现有 `showDialog` 弹窗逻辑（banner 不是全屏弹窗，confirm 弹窗盖在上面不违和）。

## 边界处理

| 场景 | 处理 | 依据 |
|---|---|---|
| 点右上角 X 关弹窗 | 等待确认状态清掉 | 弹窗都没了，记住没意义 |
| 点弹窗后面灰色遮罩 | 不关弹窗，等待确认状态保留 | 防止误碰；用户可能是瞥了一眼 |
| 按 ESC 键 | 什么也不发生——不绑任何动作 | 用户明说：弹窗不会被 ESC 关掉，必须点选 |
| 弹窗关掉又开 | 自动清掉 | 弹窗重新出现时是新的 |
| 切到其他页面再切回来 | banner 的等待确认状态重置 | 用户明说；全屏弹窗无法在打开时翻页，不存在该问题 |
| 弹窗开着时新提醒来 | 新弹窗排队，等当前弹窗处理完才显示 | 用户确认合理 |
| 点 banner 上另一条用药 | 等待确认切到新那条；旧那条按钮恢复原色 | 同一时刻只有一个按钮在等 |

## 无障碍（a11y）

- 键盘操作：Tab 键在 4 个按钮间切换；按一次回车 = 第一次点击；再按一次 = 第二次点击（生效）。用浏览器原生 `<button>` + `onClick` 实现，不需要特殊处理
- 屏幕阅读器：每个按钮在不同状态下要有不同的 `aria-label`：
  - 默认状态：`aria-label="已服用"`（沿用现有）
  - 等待确认状态：`aria-label="已服用，再点一次确认"`
- 焦点可见：等待确认的按钮靠背景色 + 去掉描边两个维度的变化，已经够明显，不需要额外描边

## 多语言（i18n）

- 沿用现有翻译键，不引入新的全局术语
- "等待确认"状态**不显示新文字**（颜色变化已经够明显，按钮文字保持原样如"已服用"）
- 警告文本沿用 `t('reminder.banner.skip_confirm.body')`，4 语言都有
- 需要新增的翻译键（1 个键 × 4 语言 = 4 条）：
  - `reminder.confirm.aria_pending_suffix` = "，再点一次确认"（zh / en / zh-TW / ja）
- 现有"已服用"、"跳过本次"、"计划推迟 1 天"、"计划推迟 2 天"翻译键不变

## 视觉对比度

- 主题色背景 + 原色文字：light / dark 两套主题下都要保证文字看得清。浅色主题配深色字、深色主题配浅色字都是常见组合，对比度够
- 警告文本的淡玫红底 + 玫红字：保持现有对比度（用 `var(--text-soft-rose)`）
- 改动后用 chrome-devtools 截 4 个主题 × 2 状态（默认 / 等待确认）共 8 张图，肉眼检查

## 文件改动

集中在 3 个文件：

- `src/components/ReminderModal.tsx`：
  - "已服用"按钮从 `glass-btn-primary` 降级为普通描边样式
  - 4 个按钮共享一个"等待确认标记"状态
  - "跳过本次"按钮下方增加警告文本展开区（仅 late 态）
- `src/components/ReminderBanner.tsx`：
  - "已服用"按钮从 `glass-btn-primary` 降级为普通描边样式
  - 4 个按钮共享一个"等待确认标记"状态
  - "跳过本次"按钮的 `onSkip` 回调**不**变（仍走 `showDialog` 弹窗）
- `src/i18n/translations.ts`：
  - 新增 1 个翻译键 × 4 语言（见上文"多语言"节）
- `src/components/MainLayout.tsx`（可能）：
  - 路由切换时清掉 banner 的等待确认状态（具体实现看是 prop 传 key 强制重置，还是 hook 监听路由）
  - 如果用 prop key 强制重置，ReminderBanner 加个会随路由变的 key 属性

**不修改**：

- `MainLayout.tsx` 的 `handleSkipPending`、`handleConfirmBanner`、`handleDelayPlan` 函数（按钮回调链不变）
- `useDialog`（banner 的"跳过本次"仍用它）
- `DoseFormModal.tsx`（不在本次范围）

## 实现顺序

1. 先改 `ReminderModal.tsx`（最复杂，有 on_time / late 两态 + 警告展开）
2. 改 `ReminderBanner.tsx`（去掉警告展开，复用 modal 的状态机模式）
3. 加翻译键
4. 翻页重置 banner 状态（如果是单独 hook 实现，先做；如果是 key prop，后做）
5. 4 主题 × 2 状态截图视觉验证
6. tsc + vitest

## 范围外

- 任何键盘快捷键（如 Cmd+Enter 触发等待中的按钮）—— 浏览器原生 Enter 键已经覆盖
- 长按提交 / 滑动确认等其他防错触模式 —— 用户没提，且本方案已经覆盖需求
- 撤销 snackbar / 撤销历史 —— 用户没提，且本方案不依赖"立即生效 + 撤销"
- 弹窗外的 banner 多选 / 全选 —— 与本设计无关
- DoseFormModal 的"保存 / 删除"按钮防错触 —— 单独的产品决策，本设计不涉及
